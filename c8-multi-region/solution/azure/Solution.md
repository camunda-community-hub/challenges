Every command is copy-paste ready.---## Clusters| Name | Context | Region | Namespace | regionId ||---|---|---|---|---|| East | `joey_k8s` | East US | `green-east` | 1 || West | `joey_k8s_westus2` | West US 2 | `blue-west` | 0 |## Static IPs (permanent — never change)| Service | Cluster | Static IP | Ports ||---|---|---|---|| Zeebe LB | East | `13.68.247.25` | 26501, 26502 || Zeebe LB | West | `4.154.234.246` | 26501, 26502 || Elasticsearch LB | East | `104.211.58.205` | 9200 || Elasticsearch LB | West | `4.155.211.127` | 9200 || CoreDNS Proxy LB | East | `172.212.39.11` | 53 || CoreDNS Proxy LB | West | `172.193.220.146` | 53 |## Files| File | Cluster | Purpose ||---|---|---|| `coredns-proxy-east.yaml` | East | CoreDNS proxy deployment + LB || `coredns-proxy-west.yaml` | West | CoreDNS proxy deployment + LB || `coredns-custom-east.yaml` | East | CoreDNS forwarding rules || `coredns-custom-west.yaml` | West | CoreDNS forwarding rules || `camunda-lb-east.yaml` | East | Static LB for Zeebe + Elasticsearch || `camunda-lb-west.yaml` | West | Static LB for Zeebe + Elasticsearch || `camunda-value-green.yaml` | East | Helm values for green-east || `camunda-value-blue.yaml` | West | Helm values for blue-west |---## Step 1 — Create Namespaces```shellkubectl config use-context joey_k8skubectl create namespace green-eastkubectl config use-context joey_k8s_westus2kubectl create namespace blue-west```---## Step 2 — Create Static Public IPs> Static IPs must be created in the **node resource group** (`mc_...`), not the cluster resource group.> These only need to be created ONCE. They persist forever.```shell# East - Zeebeaz network public-ip create \  --resource-group mc_joey_k8s-rg_joey_k8s_eastus \  --name camunda-zeebe-east-pip \  --sku Standard \  --allocation-method Static \  --location eastus# East - Elasticsearchaz network public-ip create \  --resource-group mc_joey_k8s-rg_joey_k8s_eastus \  --name camunda-es-east-pip \  --sku Standard \  --allocation-method Static \  --location eastus# West - Zeebeaz network public-ip create \  --resource-group mc_joey_k8s_westus2-rg_joey_k8s_westus2_westus2 \  --name camunda-zeebe-west-pip \  --sku Standard \  --allocation-method Static \  --location westus2# West - Elasticsearchaz network public-ip create \  --resource-group mc_joey_k8s_westus2-rg_joey_k8s_westus2_westus2 \  --name camunda-es-west-pip \  --sku Standard \  --allocation-method Static \  --location westus2```Get and verify the IPs:```shellaz network public-ip show --resource-group mc_joey_k8s-rg_joey_k8s_eastus --name camunda-zeebe-east-pip --query ipAddress -o tsv# → 13.68.247.25az network public-ip show --resource-group mc_joey_k8s-rg_joey_k8s_eastus --name camunda-es-east-pip --query ipAddress -o tsv# → 104.211.58.205az network public-ip show --resource-group mc_joey_k8s_westus2-rg_joey_k8s_westus2_westus2 --name camunda-zeebe-west-pip --query ipAddress -o tsv# → 4.154.234.246az network public-ip show --resource-group mc_joey_k8s_westus2-rg_joey_k8s_westus2_westus2 --name camunda-es-west-pip --query ipAddress -o tsv# → 4.155.211.127```---## Step 3 — Create NSG Rules> These only need to be created ONCE. clusters. We deploy a dedicated CoreDNS proxy on each cluster.```shell# East clusterkubectl config use-context joey_k8skubectl apply -f coredns-proxy-east.yaml# West clusterkubectl config use-context joey_k8s_westus2kubectl apply -f coredns-proxy-west.yaml```Wait for proxy LB IPs to be assigned:```shellkubectl config use-context joey_k8skubectl get svc coredns-proxy-lb -n kube-system# EXTERNAL-IP should be: 172.212.39.11kubectl config use-context joey_k8s_westus2kubectl get svc coredns-proxy-lb -n kube-system# EXTERNAL-IP should be: 172.193.220.146```Verify CoreDNS proxy is running:```shellkubectl config use-context joey_k8skubectl logs -n kube-system -l app=coredns-proxy --tail=10kubectl config use-context joey_k8s_westus2kubectl logs -n kube-system -l app=coredns-proxy --tail=10```---## Step 5 — Configure CoreDNS Forwarding```shell# East cluster - Always run Step 6's ClusterIP check after each fresh install and update the yaml files before pods become ready.### ⚠️ Never use FQDN for local ElasticsearchDo NOT use `camunda-elasticsearch.green-east.svc.cluster.local:9200` for local connections. AKS CoreDNS proxy causes `REFUSED` errors for own-cluster FQDN. Use the ClusterIP directly (e.g. `10.0.35.229:9200`).### ⚠️ Never delete LB servicesAlways use `kubectl apply` to update LB services. Deleting them causes Azure to release the static IP temporarily, requiring a redeploy.### ⚠️ Zeebe brokers not finding each otherCheck the advertised IPs match the static LB IPs:```shellkubectl exec -n green-east camunda-zeebe-0 -- env | grep ADVERTISED# ZEEBE_BROKER_NETWORK_ADVERTISEDHOST=13.68.247.25 ✅# ZEEBE_BROKER_NETWORK_COMMANDAPI_ADVERTISEDHOST=13.68.247.25 ✅kubectl exec -n blue-west camunda-zeebe-0 -- env | grep ADVERTISED# ZEEBE_BROKER_NETWORK_ADVERTISEDHOST=4.154.234.246 ✅# ZEEBE_BROKER_NETWORK_COMMANDAPI_ADVERTISEDHOST=4.154.234.246 ✅```If wrong, run helm upgrade with correct yaml then:```shellkubectl rollout restart statefulset camunda-zeebe -n green-eastkubectl rollout restart statefulset camunda-zeebe -n blue-west```---## Why We Use IP Addresses Instead of DNS Names### How GKE Does It (Simpler)In GKE, the Zeebe contact points use internal DNS names directly:```yaml- name: ZEEBE_BROKER_CLUSTER_INITIALCONTACTPOINTS  value: "camunda-zeebe-0.camunda-zeebe.green-east.svc.cluster.local:26502,          camunda-zeebe-1.camunda-zeebe.green-east.svc.cluster.local:26502,          camunda-zeebe-0.camunda-zeebe.blue-west.svc.cluster.local:26502,          camunda-zeebe-1.camunda-zeebe.blue-west.svc.cluster.local:26502"```This works in GKE because:1. GKE uses **VPC-native networking** — pod IPs are directly routable across clusters within the same VPC2. `kube-dns` stub domains forward `green-east.svc.cluster.local` to the other cluster's DNS LB3. The DNS LB resolves the name and returns a **pod IP** that is directly reachable### Why This Doesn't Work in AKSAKS with **Azure CNI Overlay** mode assigns pod IPs from a private overlay network (`10.244.0.0/16`). These pod IPs are:- Only routable **within the same cluster**- **Not routable** between clusters, even in the same Azure regionSo when GKE resolves `camunda-zeebe-0.camunda-zeebe.blue-west.svc.cluster.local` it gets back `10.92.0.4` — a VPC-routable IP. When AKS resolves the same name it gets back `10.244.0.x` — a pod overlay IP that the other cluster cannot reach.This is why even after getting DNS to resolve correctly, we still saw:```No route to host: camunda-zeebe-0.camunda-zeebe.blue-west.svc:26502 (10.244.0.247:26502)```### What We Had to Do InsteadWe had to:1. Deploy **LoadBalancer services** to expose Zeebe and Elasticsearch externally via public IPs2. Tell each Zeebe broker to **advertise its public LB IP** instead of its internal pod IP3. Use **static public IPs** so the addresses never change between deployments```yaml# Each broker advertises its own public LB IP- name: ZEEBE_BROKER_NETWORK_ADVERTISEDHOST  value: "13.68.247.25"  # East cluster's static Zeebe LB IP- name: ZEEBE_BROKER_NETWORK_COMMANDAPI_ADVERTISEDHOST  value: "13.68.247.25"# Contact points use public LB IPs instead of DNS names- name: ZEEBE_BROKER_CLUSTER_INITIALCONTACTPOINTS  value: "13.68.247.25:26502,4.154.234.246:26502"```---## Ideal Production Setup on AKSIf you want to achieve the same simplicity as GKE with DNS names, there are two recommended approaches:### Option 1 — VNet Peering (Best — mirrors GKE architecture)Peer the two AKS cluster VNets together. This makes pod IPs directly routable between clusters, eliminating the need for LoadBalancers entirely.```shell# Get VNet IDsEAST_VNET=$(az aks show --resource-group joey_k8s-rg --name joey_k8s \  --query networkProfile.networkPlugin -o tsv)# Peer east → westaz network vnet peering create \  --name east-to-west \  --resource-group mc_joey_k8s-rg_joey_k8s_eastus \  --vnet-name <east-vnet-name> \  --remote-vnet <west-vnet-id> \  --allow-vnet-access# Peer west → eastaz network vnet peering create \  --name west-to-east \  --resource-group mc_joey_k8s_westus2-rg_joey_k8s_westus2_westus2 \  --vnet-name <west-vnet-name> \  --remote-vnet <east-vnet-id> \  --allow-vnet-access```With VNet peering you can go back to the simple GKE-style config:```yaml- name: ZEEBE_BROKER_CLUSTER_INITIALCONTACTPOINTS  value: "camunda-zeebe-0.camunda-zeebe.green-east.svc.cluster.local:26502,          camunda-zeebe-0.camunda-zeebe.blue-west.svc.cluster.local:26502"```**Pros:** Simplest config, no public IPs needed, mirrors GKE exactly**Cons:** Requires non-overlapping pod CIDRs between clusters, VNet peering has bandwidth costs### Option 2 — Azure DNS + Static IPs (What We Use)Use Azure DNS to give friendly hostnames to the static IPs instead of raw IPs in the config.```shell# Create a DNS zoneaz network dns zone create \  --resource-group joey_k8s-rg \  --name camunda.internal.example.com# Add DNS recordsaz network dns record-set a add-record \  --resource-group joey_k8s-rg \  --zone-name camunda.internal.example.com \  --record-set-name zeebe-east \  --ipv4-address 13.68.247.25az network dns record-set a add-record \  --resource-group joey_k8s-rg \  --zone-name camunda.internal.example.com \  --record-set-name zeebe-west \  --ipv4-address 4.154.234.246```Then use DNS names in your config instead of IPs:```yaml- name: ZEEBE_BROKER_CLUSTER_INITIALCONTACTPOINTS  value: "zeebe-east.camunda.internal.example.com:26502,          zeebe-west.camunda.internal.example.com:26502"```**Pros:** Human readable, IPs can change without updating config, easy to remember**Cons:** Still uses public IPs, DNS propagation delay### Summary| Approach | Complexity | Security | Cost | GKE Parity ||---|---|---|---|---|| GKE VPC-native | Simple | High (private) | Low | ✅ Identical || AKS VNet Peering | Medium | High (private) | Medium | ✅ Identical || AKS Static IPs + DNS | Medium | Low (public) | Medium | ⚠️ Similar || AKS Static IPs (current) | High | Low (public) | Medium | ❌ Different |For production, **VNet Peering** is the recommended approach on AKS as it matches the GKE architecture most closely and keeps all traffic private.
